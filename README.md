# Требования к ПО

Разработка любого программного продукта начинается с процесса формулирования требований к ПС, в котором, исходя из довольно смутных пожеланий заказчика, результатов исследования бизнес-процессов и личного опыта разработчика должен быть создан документ, максимально точно определяющий задачи разработчиков ПС.

__Требование — это утверждение, которое идентифицирует эксплуатационные, функциональные параметры, характеристики или ограничения проектирования продукта или процесса, которое однозначно, проверяемо и измеримо [ISO/IEC 29148]__

**Источники**

- Наблюдение за работой преподавателей в ходе лекций
- Опросы студентов об удобстве получения материала
- Анализ существующих систем трансляции контента

# Методы сбора и извлечения требований

## Анализ предметной области

Программное решение реализует модель предметной области, поэтому сначала ее нужно построить, а для этого необходимо понять, осмыслить и проанализировать бизнес-требования к программной системе.

**Источники требований**

- беседы с преподавателями и студентами;
- анализ существующих инструментов для проведения лекций (Mentimeter, Slido, Teams);
- наблюдение за процессом проведения лекций в аудитории;
- анализ запросов по улучшению взаимодействия лектор–студент.

## Задача 1. Концепция проекта

### 1. Начните с описания проблемы

*Проблема — отсутствие удобного инструмента, позволяющего преподавателю в режиме реального времени транслировать слайды презентации студентам и организовывать обратную связь непосредственно во время лекции, не прерывая её ход.*

*Воздействует на — преподавателей, проводящих лекции с использованием PowerPoint; студентов, участвующих в лекции очно или дистанционно.*

*Результатом чего является — студенты не могут следить за слайдами при плохой видимости экрана; нет удобного способа задавать вопросы, не перебивая лектора; преподаватель не видит, насколько материал понятен аудитории.*

*Выигрыш от ПП — мгновенная доставка слайдов студентам через Telegram-бот; возможность задавать вопросы анонимно прямо в чате; аналитика вопросов по лекции для преподавателя.*

### 2. Определите назначение программного средства

*Какие системы смогут применять ПС для решения своих задач?*
Учебные заведения, проводящие лекционные занятия с использованием презентаций PowerPoint.

*От каких систем/пользователей поступают входные данные ПС?*
- Преподаватель — управляет лекцией через надстройку PowerPoint (запуск, смена слайда, завершение)
- Студент — подключается к лекции и задаёт вопросы через Telegram-бот

*Кто/что использует результаты?*
- Студенты получают изображения слайдов в Telegram в реальном времени
- Преподаватель видит список вопросов и может отвечать на них

*Какие сторонние системы/сервисы участвуют в процессе обработки данных?*
- Telegram Bot API — канал уведомлений и вопросов
- Microsoft PowerPoint — среда работы преподавателя
- Office Add-in (JavaScript) — встроенная панель управления лекцией

### 3. Контекстная диаграмма

```
┌────────────────┐        HTTPS/WS        ┌─────────────────────────┐
│   PowerPoint   │◄──────────────────────►│   PresAssistant Server  │
│  (Add-in)      │   REST + WebSocket     │   Spring Boot + H2      │
└────────────────┘                        └───────────┬─────────────┘

                                                      │
                                          ┌───────────▼─────────────┐
                                          │    Telegram-клиент      │
                                          │    (студент)            │
                                          └─────────────────────────┘
```

## Составление словаря предметной области

| Термин | Определение |
|--------|-------------|
| **Лекция** | Учебное занятие с презентацией, в рамках которого преподаватель транслирует слайды студентам |
| **Сессия лекции** | Активный сеанс между запуском и завершением лекции преподавателем |
| **Слайд** | Одна страница презентации PowerPoint, передаваемая студентам в виде изображения |
| **Надстройка (Add-in)** | Встроенная в PowerPoint панель управления, реализованная как Office Web Add-in на JavaScript |
| **Студент** | Пользователь Telegram-бота, подключившийся к активной лекции по названию |
| **Лектор** | Преподаватель, управляющий сессией через надстройку PowerPoint |
| **Telegram-бот** | Бот, выступающий каналом связи между сервером и студентами |
| **Вопрос** | Текстовое сообщение студента, отправленное в бот во время активной лекции |
| **Трансляция** | Автоматическая рассылка изображения текущего слайда всем подключённым студентам |

## Задача 3. Анализ конкурентов

Сравнительная характеристика конкурирующих решений:

| Проблема | Сегмент рынка | Mentimeter | Slido | MS Teams | PresAssistant | Приоритет |
|----------|--------------|------------|-------|----------|---------------|-----------|
| Трансляция слайдов студентам в реальном времени | U, XS, S | - | - | + | + | e |
| Встроенная интеграция с PowerPoint | U, XS | - | + | + | + | e |
| Приём вопросов от студентов без регистрации | U, XS, S | + | + | - | + | u |
| Работа через мессенджер (Telegram) | U | - | - | - | + | u |
| Автономная работа без облачного сервиса | XS | - | - | - | + | d |
| Отображение аналитики вопросов | S, M | + | + | + | + | u |
| Работа без VPN и корпоративных аккаунтов | U, XS | + | - | - | + | e |

**Вывод:** Основные конкуренты (Slido, Teams) требуют корпоративных аккаунтов и интернет-зависимой инфраструктуры. PresAssistant выделяется интеграцией с Telegram, доступным без регистрации студентам, и возможностью автономной работы.

---

### Создание образа программного продукта

*(Шаблон Moore, 1991)*

**«для»** — преподавателей высших и средних учебных заведений, проводящих лекции в PowerPoint;

**«который»** — нуждаются в простом способе доставки слайдов студентам и сборе вопросов в реальном времени;

**«этот»** — Ассистент лектора (PresAssistant);

**«является»** — системой интерактивного сопровождения лекций;

**«который»** — позволяет транслировать слайды и получать вопросы через Telegram без регистрации студентов;

**«в отличие от»** — Slido и Microsoft Teams, требующих корпоративных аккаунтов и настройки IT-инфраструктуры;

**«наш продукт»** — работает локально, устанавливается одним файлом, не требует от студентов ничего кроме Telegram.

## Задача 5. Моделирование сценариев использования

### 1. Иерархия заинтересованных лиц

```
Пользователи PresAssistant
├── Лектор
│   ├── Запуск лекции
│   ├── Смена слайда
│   ├── Трансляция слайда студентам
│   ├── Просмотр вопросов
│   └── Завершение лекции
└── Студент
    ├── Подключение к лекции
    ├── Получение слайдов
    └── Задать вопрос
```

### 2. Варианты использования (Use Cases)

**Актор: Лектор**

| № | Цель | Сценарий |
|---|------|---------|
| UC-1 | Запустить лекцию | Открывает PowerPoint → панель надстройки → вводит название → нажимает «Начать лекцию» |
| UC-2 | Транслировать слайд | Переходит на новый слайд → нажимает «Отправить слайд» → студенты получают изображение в Telegram |
| UC-3 | Просмотреть вопросы | В панели надстройки открывает список вопросов студентов |
| UC-4 | Завершить лекцию | Нажимает «Завершить лекцию» → бот уведомляет всех студентов |

**Актор: Студент**

| № | Цель | Сценарий |
|---|------|---------|
| UC-5 | Подключиться к лекции | Открывает Telegram-бот → отправляет название лекции → получает подтверждение |
| UC-6 | Получить слайд | Автоматически — бот присылает изображение при каждой смене слайда лектором |
| UC-7 | Задать вопрос | Отправляет текстовое сообщение боту во время лекции |

### 3. Перечень пользовательских функциональных требований

| Источник требований | Приоритет | Характеристика | Описание |
|--------------------|-----------|----------------|----------|
| Лектор | e | Запуск сессии | Система должна позволять создавать именованную сессию лекции |
| Лектор | e | Трансляция слайдов | Система должна отправлять изображение текущего слайда всем подключённым студентам |
| Лектор | u | Просмотр вопросов | Система должна отображать список вопросов от студентов в панели надстройки |
| Студент | e | Подключение через Telegram | Студент должен подключиться к лекции, зная только её название |
| Студент | e | Получение слайдов | Студент должен автоматически получать слайды без дополнительных действий |
| Студент | u | Отправка вопроса | Студент должен иметь возможность отправить текстовый вопрос во время лекции |
| Оба | d | Уведомление о завершении | Все студенты должны получить уведомление об окончании лекции |

## Задача 7. Составление технического задания

### Цель разработки
Разработать программную систему «Ассистент лектора» для интерактивного сопровождения лекций с трансляцией слайдов и сбором вопросов через Telegram.

### Требования к функциональности

**Сервер (Spring Boot):**
- Управление сессиями лекций (создание, завершение)
- Преобразование слайдов `.pptx` в изображения (Apache POI)
- Рассылка изображений через Telegram Bot API
- REST API для надстройки PowerPoint
- WebSocket-уведомления о смене слайда

**Надстройка PowerPoint (Office Add-in):**
- Панель управления лекцией (запуск, смена слайда, завершение)
- Отображение списка вопросов студентов
- Интеграция с сервером по HTTPS

**Telegram-бот:**
- Приём команды подключения к лекции по названию
- Рассылка изображений слайдов подключённым студентам
- Приём текстовых вопросов во время лекции

### Требования к развёртыванию
- Установка на Windows без предустановленной Java
- Работа на `localhost:8082` с самоподписанным SSL-сертификатом
- H2-база данных (встроенная, без отдельной СУБД)
- Установщик с автоматической регистрацией надстройки и сертификата

## Задача 8. Составление спецификации ПО

### Функциональные требования

| ID | Требование | Приоритет | Верификация |
|----|-----------|-----------|-------------|
| FR-1 | Сервер запускает сессию лекции с уникальным названием | e | Тест: POST /api/lectures → 200 OK |
| FR-2 | При смене слайда система рассылает его изображение всем студентам | e | Тест: студент получает фото в Telegram |
| FR-3 | Студент подключается к лекции командой с названием в боте | e | Тест: /start + название → подтверждение |
| FR-4 | Вопрос студента отображается в надстройке лектора | u | Тест: сообщение бота → список в панели |
| FR-5 | При завершении лекции все студенты получают уведомление | u | Тест: POST /api/lectures/{id}/end → Telegram |

### Нефункциональные требования

| ID | Требование | Категория |
|----|-----------|-----------|
| NFR-1 | Время доставки слайда студентам — не более 5 секунд | Производительность |
| NFR-2 | Соединение между надстройкой и сервером защищено HTTPS/TLS | Безопасность |
| NFR-3 | Приложение устанавливается без навыков разработки | Удобство эксплуатации |
| NFR-4 | База данных не требует отдельной установки СУБД | Простота развёртывания |
| NFR-5 | Система работает при отсутствии интернета (кроме Telegram API) | Надёжность |

### Архитектурные ограничения
- Язык реализации: Java 21, JavaScript (ES6+)
- Фреймворк: Spring Boot 4, Office Add-in API
- База данных: H2 (embedded), Flyway для миграций
- Архитектурный стиль: Hexagonal Architecture (Ports & Adapters)
- Протоколы: HTTPS (REST), WebSocket (STOMP), Telegram Bot API

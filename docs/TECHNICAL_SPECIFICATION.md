# Техническое задание
## Программная система «Ассистент лектора» (PresAssistant)

**ГОСТ 34.602-89 / IEEE 830 / ISO/IEC 29148**

---

## 1. Введение

### 1.1 Цель документа

Настоящее техническое задание определяет цели, функциональные и нефункциональные требования, ограничения и критерии приёмки программной системы «Ассистент лектора» (PresAssistant), предназначенной для интерактивного сопровождения лекционных занятий.

### 1.2 Область применения

Система предназначена для учебных заведений — университетов, колледжей, школ, — проводящих лекционные занятия с использованием Microsoft PowerPoint. Применима как при очном, так и при смешанном формате (часть студентов присутствует дистанционно).

### 1.3 Определения и сокращения

| Термин | Определение |
|--------|-------------|
| **Лекция** | Учебное занятие с презентацией, в рамках которого преподаватель транслирует слайды студентам |

## 2. Концепция проекта

### 2.1 Описание проблемы

**Проблема** — отсутствие удобного инструмента, позволяющего преподавателю в режиме реального времени уведомлять студентов о текущем слайде презентации и организовывать обратную связь непосредственно во время лекции, не прерывая её ход.

**Воздействует на** — преподавателей высших и средних учебных заведений, проводящих лекции с использованием PowerPoint; студентов, участвующих в лекции очно или дистанционно.

**Результатом чего является:**
- студенты не могут следить за слайдами при плохой видимости экрана или дистанционном участии;
- нет удобного способа задавать вопросы, не перебивая лектора;
- преподаватель не видит, насколько материал понятен аудитории и кто активно следит за лекцией.

**Выигрыш от ПП:**
- мгновенное уведомление студентов о смене слайда с возможностью запросить его изображение через Telegram-бот;
- возможность задавать вопросы анонимно прямо в чате без прерывания лекции;
- аналитика активности студентов и история вопросов для преподавателя.

---

### 2.2 Назначение программного средства

**Какие системы смогут применять ПС?**
Учебные заведения, проводящие лекционные занятия с использованием Microsoft PowerPoint. Система применима для очного формата (все студенты в аудитории) и смешанного (часть участвует дистанционно).

**От каких систем/пользователей поступают входные данные ПС?**
- **Лектор** — создаёт сессию, управляет слайдами, завершает лекцию через надстройку PowerPoint
- **Студент** — подключается к лекции и запрашивает слайды через Telegram-бот
- **Microsoft PowerPoint** — источник событий смены слайда, предоставляет SVG через Office.js API

**Кто/что использует результаты?**
- **Студент** — получает уведомления о текущем слайде и изображения по запросу в Telegram
- **Лектор** — видит список вопросов, состав аудитории, аналитику запросов слайдов
- **Telegram** — транспортный канал доставки уведомлений и изображений

**Сторонние системы/сервисы:**
- **Telegram Bot API** — двустороннее взаимодействие: уведомления, изображения, вопросы
- **Microsoft PowerPoint / Office Add-in API** — среда работы лектора, источник событий
- **Windows Certificate Store** — хранилище доверенных SSL-сертификатов

### 2.4 Образ продукта (шаблон Moore, 1991)

**«для»** — преподавателей высших и средних учебных заведений, проводящих лекции в Microsoft PowerPoint;

**«который»** — нуждаются в простом способе информировать студентов о текущем слайде и собирать их вопросы в режиме реального времени без прерывания хода занятия;

**«этот»** — Ассистент лектора (PresAssistant);

**«является»** — системой интерактивного сопровождения лекций, встроенной в PowerPoint;

**«который»** — автоматически уведомляет студентов о смене слайда, позволяет им запросить его изображение через Telegram-бот и собирает вопросы аудитории без регистрации;

**«в отличие от»** — Slido и Microsoft Teams, требующих корпоративных аккаунтов, настройки IT-инфраструктуры и установки дополнительного ПО студентами;

**«наш продукт»** — работает полностью локально, устанавливается одним MSI-файлом, не требует от студентов ничего, кроме Telegram, и не передаёт данные лекций сторонним облакам.

## 3. Пользователи системы

### 3.1 Иерархия заинтересованных лиц (метод опорных точек зрения)

```
Пользователи PresAssistant
├── Лектор
│   ├── Управляет сессией лекции
│   ├── Транслирует уведомления о слайдах
│   ├── Работает с вопросами аудитории
│   ├── Управляет составом студентов
│   └── Завершает лекцию
├── Студент
│   ├── Подключается к лекции через Telegram
│   ├── Запрашивает изображение слайда
│   └── Задаёт вопросы лектору
└── IT-Администратор
    ├── Устанавливает систему (MSI)
    ├── Настраивает SSL-сертификат
    ├── Регистрирует надстройку в Office
    └── Настраивает токен Telegram-бота
```

| Роль | Потребности | Технический уровень |
|------|------------|---------------------|
| **Лектор** | Простое управление лекцией без переключения между приложениями | Средний (умеет пользоваться надстройками Office) |
| **Студент** | Получать слайды и задавать вопросы без регистрации и установки приложений | Базовый (умеет пользоваться Telegram) |
| **IT-Администратор** | Однократная установка, без сложной конфигурации | Выше среднего (SSL, реестр Windows, PowerShell) |

### 3.2 Сценарии использования

#### UC-1: Начать лекцию
**Актор:** Лектор  
**Предусловие:** Сервер запущен; надстройка открыта в PowerPoint; презентация открыта.  
**Основной поток:**
1. Лектор вводит название лекции и выбирает режим (анонимный / поимённый)
2. Нажимает «Начать лекцию»
3. Система создаёт сессию в БД, возвращает UUID
4. Надстройка генерирует QR-код и отображает экран активной лекции

**Альтернативный поток А1:** Сервер недоступен → система показывает ошибку.  
**Альтернативный поток А2:** Название не введено → система блокирует запуск.  
**Постусловие:** Сессия активна; студенты могут подключиться по QR-коду.

---

#### UC-2: Транслировать уведомление о слайде
**Актор:** Лектор  
**Предусловие:** Лекция активна (UC-1 выполнен).  
**Основной поток:**
1. Лектор переходит на следующий слайд
2. Office.js фиксирует событие смены слайда
3. Надстройка захватывает номер и SVG-изображение слайда
4. SVG конвертируется в PNG, отправляется на сервер
5. Сервер рассылает уведомление с номером слайда всем студентам

**Альтернативный поток А1:** Ручной режим → лектор нажимает кнопку «Отправить» вручную.  
**Постусловие:** Студенты получили уведомление; изображение сохранено на сервере.

---

#### UC-3: Просмотреть и удалить вопрос
**Актор:** Лектор  
**Предусловие:** Лекция активна; есть хотя бы один вопрос.  
**Основной поток:**
1. Лектор открывает вкладку «Вопросы»
2. Система отображает список вопросов (новые — сверху)
3. Лектор читает вопрос, отвечает устно, нажимает «✕»
4. Вопрос удаляется из списка и БД

**Постусловие:** Вопрос удалён; счётчик уменьшился.

---

#### UC-4: Подключиться к лекции
**Актор:** Студент  
**Предусловие:** Лекция активна; у студента установлен Telegram.  
**Основной поток:**
1. Студент сканирует QR-код
2. Telegram открывается с командой `/start {lectureId}`
3. Бот регистрирует студента в БД и отправляет подтверждение

**Альтернативный поток А1:** Поимённый режим → бот запрашивает Ф.И.О.  
**Альтернативный поток А2:** Лекция завершена → бот сообщает об этом.  
**Постусловие:** Студент зарегистрирован; будет получать уведомления о слайдах.

---

#### UC-5: Запросить изображение слайда
**Актор:** Студент  
**Предусловие:** Студент подключён; лектор показал хотя бы один слайд.  
**Основной поток:**
1. Студент получает уведомление «Лектор перешёл на слайд N»
2. Нажимает кнопку «Получить слайд» или отправляет `/slide`
3. Сервер возвращает PNG-изображение текущего слайда

**Альтернативный поток А1:** Изображение недоступно → бот сообщает «Слайд пока недоступен».  
**Постусловие:** Студент получил изображение в чате.

---

#### UC-6: Задать вопрос
**Актор:** Студент  
**Предусловие:** Студент подключён к активной лекции.  
**Основной поток:**
1. Студент отправляет текстовое сообщение боту
2. Система сохраняет вопрос с привязкой к сессии и текущему слайду
3. Бот подтверждает: «Вопрос отправлен лектору»
4. Вопрос появляется в панели лектора

**Альтернативный поток А1:** Студент выгнан → сообщение игнорируется.  
**Постусловие:** Вопрос в БД и в панели лектора.

---

#### UC-7: Установить систему
**Актор:** IT-Администратор  
**Предусловие:** Windows 10/11; Microsoft Office 2016+; права администратора.  
**Основной поток:**
1. Запустить `PresAssistant-1.0.0.msi`, выбрать папку
2. Открыть лаунчер, ввести токен бота
3. Нажать «Запустить сервер»
4. Нажать «Установить SSL-сертификат» (UAC-диалог)
5. Нажать «Зарегистрировать надстройку»
6. Перезапустить PowerPoint

**Альтернативный поток А1:** UAC отключён → скрипт запускается напрямую.  
**Постусловие:** Система готова к работе; надстройка доступна в PowerPoint.

## 5. Нефункциональные требования

### 5.1 Производительность (Performance Efficiency — ISO 25010)

| ID | Требование | Метрика |
|----|-----------|---------|
| NFR-01 | Время доставки уведомления о слайде студентам | ≤ 5 секунд при стабильном интернете |
| NFR-02 | Автообновление списка вопросов | не реже 1 раза в 10 секунд |
| NFR-03 | Потребление памяти сервером в режиме ожидания | ≤ 100 МБ RAM |

### 5.2 Надёжность (Reliability)

| ID | Требование |
|----|-----------|
| NFR-04 | При недоступности Telegram Bot API система не должна аварийно завершаться — ошибки логируются |
| NFR-05 | В режиме SlideShow надстройка переходит в polling-режим (каждые 800 мс) для отслеживания смены слайда |

### 5.3 Безопасность (Security)

| ID | Требование |
|----|-----------|

| **Четкость** | Есть ли требования с союзом «и» (не атомарные)? | ✓ | Проверено в 7.2 |

### 7.2 Проверка на атомарность

| ID | Исходная формулировка | Статус | Разбиение |
|----|----------------------|--------|-----------|
| FR-04, FR-05 | «Отправлять слайд автоматически и вручную» | ⚠ Разбито | FR-04: автоматически; FR-05: вручную |
| FR-07, FR-08 | «Просматривать и удалять вопросы» | ⚠ Разбито | FR-07: просмотр; FR-08: удаление |
| FR-15, FR-16 | «Установить сертификат и надстройку» | ⚠ Разбито | FR-15: сертификат; FR-16: надстройка |

### 7.3 Проверка на непротиворечивость

| Пара требований | Противоречие? | Обоснование |
|----------------|---------------|-------------|
| NFR-06 (HTTPS) и NFR-10 (простая установка) | Нет | HTTPS реализован через автоматическую установку сертификата в лаунчере |
| NFR-01 (≤5 сек) и NFR-06 (шифрование) | Нет | SSL-шифрование добавляет <50 мс накладных расходов, укладывается в 5 сек |
| FR-04 (авто-рассылка) и FR-05 (ручной режим) | Нет | Взаимоисключающие режимы, управляются переключателем |

### 7.4 Проверка на полноту

| Аспект | Покрыт? | Комментарий |
|--------|---------|-------------|
| Все роли описаны? | ✓ | Лектор, Студент, IT-Администратор |
| Крайние случаи учтены? | ✓ | Альтернативные потоки в UC-1÷7 |
| Требования к обработке ошибок? | ✓ | NFR-04; альтернативные потоки UC |
| Поведение при сбоях? | ✓ | NFR-04 (Telegram API); NFR-05 (SlideShow) |
| Требования к безопасности? | ✓ | NFR-06÷09 |
| Требования к установке? | ✓ | FR-15÷17, NFR-10÷11 |

### 7.5 SWOT-анализ спецификации

| Квадрант | Оценка |
|----------|--------|
| **Strengths** | Полное покрытие всех ролей; метрики в NFR; трассируемость через UC → FR; атомарные требования |
| **Weaknesses** | REST API не защищён аутентификацией (любой, знающий UUID лекции, может обратиться к API); CORS разрешает все источники |
| **Opportunities** | Добавить JWT-аутентификацию; расширить аналитику; поддержка Google Slides |
| **Threats** | Изменения Telegram Bot API; устаревание Office Add-in API; блокировка Telegram в сети учреждения |
